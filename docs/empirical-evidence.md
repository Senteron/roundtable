# Empirical evidence behind Roundtable's design

This document is the audit trail for the empirical claims that appear in
[`design.md`](design.md) and [`decisions.md`](decisions.md). Several of
Roundtable's load-bearing design choices — no vote-to-continue, no
synthesis stage, no persistence, N-1 panel tolerance, the latency and
cost budgets that informed the timeout defaults — are justified in
those documents with phrases like "the corpus showed" or "the empirical
data is unambiguous." This document is *the corpus* and *the data*.

Read this when:

- You are about to add a vote/continue/converge mechanism back to the
  tool, on the assumption that "models can vote sensibly if you ask
  the right way."
- You are about to weaken the no-persistence invariant, on the
  assumption that "users will want runs back later."
- You are about to add a synthesis stage, on the assumption that "the
  panel needs a unified answer."
- You are about to lower or raise the per-call timeout default.
- A reviewer asks "where did the round-cap recommendation come from?"

If you have not yet read [`decisions.md`](decisions.md), read that
first. This document is the receipts; that document is the verdict.

## 1. Data sources

The Roundtable design draws on three corpora, all from Senteron (the
predecessor CLI tool) on the same author's machine and prompts.

### 1.1 The "internal" corpus (primary)

Location (off-repo, on author's machine):
`~/.../crucible-internal/runs/`

Contents at time of analysis (May 2026):

| File type | Count | Use |
|---|---|---|
| `crucible_*.json` (real runs, `pipeline == "peer_review"`) | **67** | Primary evidence |
| `crucible_*.json` (real runs, `pipeline == "judge_only"`) | 26 | Latency comparison only |
| `crucible_*.json` (legacy, no `pipeline` field) | 3 | Excluded from analysis |
| `*.checkpoint` | 151 | In-progress run state; not analyzed |
| `*.events` | 128 | Stage-event traces |
| `events.db` (SQLite) | 1 | 202 run records, 6109 events, 638 model_run_stats; cross-check |

Date range: **2026-02-14 → 2026-05-22** (14 weeks).
Unique prompt prefixes (first 100 chars): **59 of 67** (8 retries on
the same prompt; mostly debugging iterations).
Single author, single machine, prompts drawn from real personal
workflow (mix of professional and technical questions; specific
prompt content is not committed to the repo).

### 1.2 The "archive" corpus (pre-peer-review baseline)

Location: `~/.../crucible-old/runs/`. **11 real
runs**, all single-shot judge-only, all from Feb 9–13, 2026. Schema
predates `head_to_head_rounds` and the vote fields, so this corpus
cannot speak to convergence behavior. Used here only as the source of
the 4 prompts re-run for the fresh comparison in §1.3.

### 1.3 The fresh comparison runs (added 2026-05-23/24)

Generated specifically to validate that the corpus findings still hold
on the current Senteron build, against a default 3-model panel
(`claude` / `gpt` / `gemini`). Five runs:

| File | Prompt size | Wall (s) | Cost ($) | Rounds | Stop votes |
|---|---|---|---|---|---|
| `senteron_20260523_203234.json` | tiny advice (61 chars) | 105 | 0.049 | 1 | 2/2 |
| `senteron_20260524_080451.json` | tiny advice (61 chars) | 216 | 0.087 | 1 | 2/2 |
| `senteron_20260524_080828.json` | professional CV (1.5k) | 264 | 0.240 | 1 | 2/2 |
| `senteron_20260524_081252.json` | long technical README (7.9k) | 376 | 0.336 | 1 | 2/2 |
| `senteron_20260524_081908.json` | code review (17k) | 244 | 0.439 | 1 | 2/2 |

These runs live in the Senteron working copy's `runs/` directory and
are not committed to either repo (Senteron's `runs/` is gitignored;
Roundtable doesn't have a `runs/` at all).

### 1.4 Combined dataset for the rest of this document

When this document says "72 runs" or "146 votes," it means **the 67
internal corpus runs plus the 5 fresh runs**, treated as a single
peer-review dataset. The 67 are corpus-historical; the 5 are
verification.

### 1.5 What the corpus does NOT cover

- **Other users.** All 72 runs were generated by a single user. Vote
  behavior, round count, and prompt mix may differ for other users.
- **Other orchestrators.** Senteron's pipeline uses a judge (`o3` in
  most runs), not Claude-as-orchestrator. The orchestration question
  Roundtable is designed around — *can Opus hold a thesis across
  multiple critique rounds without regressing* — is **not** answered
  by this corpus. It is answered by the two live tests recorded in
  [`decisions.md` §5](decisions.md) (voicemail and log architecture),
  not by the corpus.
- **Anthropic as a panel member.** Senteron uses Claude as a panel
  *member*, scored by the judge. Roundtable explicitly does *not*
  include Claude in the panel (Claude is the orchestrator). The
  corpus tells us about Claude's behavior when scored, not when
  orchestrating.
- **Symmetric/peer-driven loops.** Senteron's `peer_review` pipeline
  is asymmetric — the judge has authority. Roundtable's design is
  symmetric (no judge). The corpus shows what happens when models
  vote inside an asymmetric pipeline; it tells us less about a
  symmetric one. Use caution generalizing.

These caveats matter most when someone argues "the corpus says X, so
Roundtable should do Y." Check whether X actually generalizes.

## 2. Headline finding: the convergence vote was performative

This finding is the single most important input to Roundtable's design
because it justifies the most counter-intuitive decision: **Roundtable
has no vote-to-continue, no `another_round_needed` field, no
`continuation_reason` string, no panel-side convergence signal of any
kind.** [`design.md` §3](design.md) and the no-vote invariant in
[`CLAUDE.md`](../CLAUDE.md) both depend on this finding.

### 2.1 Vote distribution

Across the 72 combined runs, the Senteron schema captured **146
per-model `another_round_needed` votes** (each run records a vote per
participating model per head-to-head round).

| | Continue (`True`) | Stop (`False`) |
|---|---|---|
| **67 internal-corpus runs** | 1 | 135 |
| **5 fresh comparison runs** | 0 | 10 |
| **Combined (146 votes)** | **1 (0.7%)** | **145 (99.3%)** |

The single `True` vote in the entire dataset came from one model
(`mistral`) in one run (a 10-model panel on a complex UI-design
prompt). Every other vote in every other run, by every model, was
`False`.

**Citation: the only `True` vote, ever.**

```
File: crucible_20260215_132427.json
Round: 0 (of 2 total)
Model: mistral
another_round_needed: True
continuation_reason: "The competing response introduced several
  substantive improvements that warrant integration and further
  refinement: 1. Health Score: A composite metric that provides a
  holistic view of system performance, which is more actionable than
  individual KPIs. 2. Cost per Quality Point: A single metric balancing
  cost and value, directly addressing the cost vs. quality tradeoff.
  [...] Another round would allow for deeper refinement of these
  integrated features."
```

This is the only run in the dataset that produced a second head-to-head
round. Round 2 votes were unanimous stop.

### 2.2 The boilerplate problem

Of the **114 captured continuation reasons in the corpus** (some votes
had `null` or empty reasons):

| Pattern | Count | % of reasons |
|---|---|---|
| Exact string `"No material changes expected."` | 37 | **32%** |
| Contains `"no material"` / `"no further"` / `"no additional changes"` | 91 | **80%** |
| ≤5 words total | 37 | 32% |
| Substantive (>5 words, not the exact boilerplate) | 77 | 68% |

The exact-string finding is the bombshell. **One-third of all
"reasons" the models produced were a single canned sentence.**

**Citation: the boilerplate sentence appears verbatim in at least
these runs** (spot-checked, not exhaustive):

- `crucible_20260214_115431.json` r0 `deepseek`
- `crucible_20260214_164230.json` r0 `together`
- `crucible_20260215_131904.json` r0 `gpt`
- `crucible_20260216_151220.json` r0 `grok`, `deepseek`
- `crucible_20260216_151543.json` r0 `grok`, `gemini`
- `crucible_20260216_151925.json` r0 `groq`, `deepseek_v3`
- `crucible_20260216_153541.json` r0 `groq`

The pattern is even worse than the 32% number implies: models also
prefixed *substantive* stop-votes with the boilerplate sentence, then
appended real analysis. Example:

```
File: crucible_20260214_091006.json (r0, claude, vote=False)
continuation_reason: "No material changes expected. I've integrated
  the competitor's strongest additions [substantive domain-specific
  reasoning continues for ~150 words]"
```

This signature — a fixed prefix followed by genuine reasoning — is
diagnostic of a prompt template that *instructed* the model to begin
with that phrase. The field is closer to a ritual than a signal.

### 2.3 Per-model vote behavior

| Model | Stop votes | Continue votes | % continue |
|---|---|---|---|
| `claude` | 26 | 0 | 0.0% |
| `gemini` | 33 | 0 | 0.0% |
| `gpt` | 4 | 0 | 0.0% |
| `grok` | 27 | 0 | 0.0% |
| `deepseek` | 18 | 0 | 0.0% |
| `deepseek_v3` | 12 | 0 | 0.0% |
| `groq` | 3 | 0 | 0.0% |
| `together` | 1 | 0 | 0.0% |
| **`mistral`** | **11** | **1** | **8.3%** |

Mistral is the *only* model that ever asked for another round, and
even mistral votes stop 92% of the time. **No model in this dataset
can be trusted to honestly request continuation.**

### 2.4 Why this matters for Roundtable

Roundtable's design assumes Claude (the orchestrator) decides when to
stop, not the panel. This is justified by §2.1–§2.3. If the design
ever drifts toward a panel-side stop signal, the failure mode is
predictable: 99% of runs will stop on round 1, regardless of whether
they should.

**Don't put the vote back.** If you want a "should we go another
round" signal, derive it from output diff (word-count change,
embedding distance, concept overlap), not from asking models.

### 2.5 Fresh-run corroboration

The 5 fresh runs (§1.3), generated specifically to check whether the
corpus finding still holds:

- **10/10 votes were `False`** (100% stop rate).
- **0/8 captured reasons** used the exact boilerplate sentence —
  improvement vs the corpus's 32%, but…
- **5/8 reasons (62%) still contained `"no material"` substring** —
  the linguistic pattern persists even when the exact phrasing varies.
- **2/10 reasons were empty strings** (both from `claude` on the
  largest prompts). When the model doesn't even generate a reason,
  the field carries even less signal than boilerplate.
- **2/5 runs logged `H2H JSON parse failed for claude: Unterminated
  string`** — the orchestrator-class model's vote payload sometimes
  *malforms* and needs retry. If you keep any structured-vote field
  in a future design, expect parse failures specifically on the
  highest-capability model.

The fresh data strengthens the finding rather than complicating it.

## 3. Round-count distribution

Roundtable's [`design.md` §2.1](design.md) advises Claude (in the tool
description) that "3–4 rounds is typical for complex questions; 1–2
is fine for simple ones." That advice describes what Claude-as-
orchestrator *should* allow itself to do under symmetric conditions.
The Senteron corpus describes what Senteron's *current asymmetric
pipeline* actually does.

### 3.1 What the corpus shows

| Head-to-head rounds | Run count | % |
|---|---|---|
| 1 | 71 | **98.6%** |
| 2 | 1 | 1.4% |
| 3+ | 0 | 0% |

The single 2-round run is `crucible_20260215_132427.json` (the same
run as §2.1's `True` vote — mistral asked for round 2, round 2 voted
unanimously stop).

Cross-check from `events.db`: `SELECT run_id, COUNT(*) FROM events
WHERE event_type='STAGE_START' AND stage='head_to_head' GROUP BY
run_id` returned 60 rows, distribution `{1: 60}`. Every
DB-instrumented peer-review run fired exactly one head-to-head stage.
The 2-round outlier predates the DB-event recording schema.

### 3.2 What this means and doesn't mean

It **does** mean: under Senteron's current pipeline, with a vote-based
stop signal that doesn't work (§2), the pipeline always stops on
round 1. The round-count distribution is a property of the broken
vote, not of the deliberation difficulty.

It **does not** mean: real deliberations always converge in one round.
The two live Opus 4.7 tests recorded in
[`decisions.md`](decisions.md) showed multi-round deliberations
producing genuinely improved outputs in rounds 2 and 3.

So:

- Roundtable's tool description should *not* tell Claude "most
  deliberations converge in 1 round." That would be over-fitting on
  Senteron's broken signal.
- Roundtable's tool description *should* tell Claude to stop on
  signal-density, with 3–4 rounds as the typical complex-question
  upper bound. The corpus does not contradict that; it just doesn't
  positively confirm it.

If you want positive confirmation of multi-round value, you need the
live test data, not the corpus.

## 4. Why no synthesis stage

[`decisions.md` §2](decisions.md) argues that synthesis "destroys the
thing that made the manual workflow valuable" — Claude doesn't get
confronted by raw peer answers when it reads a smoothed-over summary.
That's a UX/cognitive argument. The corpus adds two pieces of
quantitative support:

### 4.1 Synthesis is cheap to do but not cheap to consume

Synthesis stage cost across the 67 corpus runs:

- Median: $0.01 per run
- Max: $0.13 per run
- Share of total cost (median): ~5%

So synthesis isn't *expensive*, which means the argument against it
can't be "it's not worth the money." The argument is structural: it
puts a layer of averaging between Claude and the panel's actual
output. The corpus data on output averaging is in §6 below.

### 4.2 Provenance is expensive and bears no MCP value

The corpus's provenance stage (which traces ideas back to source
models) took **31.7% of median wall time** and contained one runaway
22663-second outlier (`crucible_20260310_233905.json`). For
Roundtable, where Claude is the immediate consumer, provenance
attribution has no consumer and would be pure overhead. Stripping it
is one of the larger latency wins available.

## 5. Latency budget

This section provides the empirical basis for Roundtable's per-call
timeout default (90s) and max (180s) in
[`design.md` §2.1](design.md).

### 5.1 Per-stage timing from the corpus

From `stage_timings` in 63 of 67 corpus runs that populated the field.
Numbers in seconds.

| Stage | Median | p90 | Max |
|---|---|---|---|
| `optimize` | 7.6 | 81.3 | 854.4 |
| `dispatch` (parallel proposers, round 0) | **61.1** | **182.2** | 708.9 |
| `peer_review` (scoring + revision) | 66.5 | 175.3 | 292.1 |
| `absorption` | 36.7 | 86.3 | 131.7 |
| **`head_to_head`** (one round) | **39.8** | **97.7** | **326.0** |
| `synthesis` | 26.5 | 85.3 | 305.1 |
| `provenance` | 159.6 | 415.7 | 22115.1 |

**Total wall time:** median 504s, p90 1051s, max 22663s (the
runaway), median-without-runaway 503s.

### 5.2 What this says for Roundtable

The two relevant numbers are `dispatch` (round-0 parallel calls) and
`head_to_head` (round-1+ parallel calls with peer context). Their p90
values are **182s** and **98s** respectively.

- Roundtable's `per_call_timeout_seconds` default of **90s** would
  cut into the p90 dispatch case. For round 0 with 5,000+ char
  prompts, expect occasional timeouts on the slow models
  (groq/together/cohere/grok on long inputs were the main
  contributors to the right tail).
- The **180s max** is calibrated to cover the p90 of round-0 dispatch
  while preserving the "never block forever" invariant.

The fresh runs (§1.3) tightened this picture for a **3-model panel**
(claude/gpt/gemini, no extended providers):

- Median wall: **244s** (vs. 504s for mixed 3–10-model panels)
- Per-prompt-size relationship: small (~tiny prompts) ≈ 105–216s;
  medium (1.5k) ≈ 264s; large (8–17k) ≈ 244–376s
- No run exceeded ~6 minutes

So for Roundtable's default panel (no Claude, OpenAI + Google +
DeepSeek per [`design.md` §1](design.md)), expect each round to take
**roughly 60–100s at p50 and ~180s at p90** with a properly tuned
per-call timeout.

### 5.3 The tail risk

Both corpora show that latency tails come from **specific providers**
on **long-prompt inputs**, not from prompt complexity in general.
Examples from the historical corpus:

- `crucible_20260213_091041.json`: `together` model took 1215s on a
  60k-char code-review prompt.
- `crucible_20260213_081718.json`: similar pattern, 424s.

For Roundtable v0.1, the panel is OpenAI + Google + DeepSeek per
[`design.md` §1](design.md) — none of these are the tail-risk
providers from the corpus. The 90s default timeout should be safe
for the default panel. **If a future version adds together/cohere/
groq/grok back to the default panel, revisit this number.**

## 6. Cost budget

[`design.md`](design.md) does not currently include cost guidance.
This section provides the empirical basis if and when one is needed.

### 6.1 Per-run total cost

From 67 corpus runs:

- Median: **$0.14**
- p90: **$0.34**
- Max: $0.40
- Sum across all 67 runs: **$10.69**

From 5 fresh runs (default 3-model panel):

- Median: **$0.24**
- Range: $0.05 (tiny prompt) → $0.44 (17k-char code review)

The fresh-run median is higher than the corpus median because the
corpus includes more tiny-prompt runs; the fresh runs lean toward
real prompts.

### 6.2 Per-model per-call cost

Mean / median across all corpus calls:

| Model | Mean | Median |
|---|---|---|
| `claude` | $0.0231 | $0.0206 |
| `gemini` | $0.0176 | $0.0156 |
| `gpt` | $0.0115 | $0.0108 |
| `deepseek` | $0.0032 | $0.0024 |
| `together` | $0.0013 | $0.0009 |
| `groq` | $0.0010 | $0.0007 |
| `mistral` | $0.0008 | $0.0007 |
| `cohere` | $0.0006 | $0.0004 |
| `grok` | $0.0006 | $0.0005 |
| `deepseek_v3` | $0.0004 | $0.0003 |

For Roundtable's default panel (gpt + gemini + deepseek per
[`design.md` §1](design.md)):

- Combined per-call cost: roughly `$0.011 + $0.016 + $0.003 = $0.030`
  per model-call.
- Three-model round 0: ~$0.030.
- Round 1+ adds the bundle to each call's input; the corpus's
  head_to_head stage median was $0.02 (with mixed panel sizes).
  Estimate ~$0.04–0.10 per Roundtable round depending on bundle size.
- **Three-round deliberation: expect ~$0.10–0.30 in panel costs.**

The Claude-side cost (the orchestrator) is *not* part of this number.
That's tracked separately by the Claude API/subscription and depends
on Opus pricing, not Roundtable's panel.

### 6.3 Word counts (output verbosity) by model

Median / max word count per response, from corpus round-0 dispatches:

| Model | Median | Max |
|---|---|---|
| `gemini` | **1184** | 4270 |
| `claude` | 890 | 2798 |
| `mistral` | 756 | 1882 |
| `deepseek_v3` | 655 | 3627 |
| `deepseek` | 615 | 4400 |
| `grok` | 589 | 2320 |
| `gpt` | 484 | 2079 |
| `groq` | 444 | 1241 |
| `together` | 496 | 1061 |
| `cohere` | 442 | 1840 |

**Gemini is ~2.4× more verbose than GPT at the median.** This matters
for Roundtable because bundle size grows with panel-member verbosity,
and Claude's input tokens (and the orchestrator's cost) grow with
bundle size. A gemini-heavy panel will produce a noticeably larger
context burden for the orchestrator than a gpt-heavy panel.

## 7. Why N-1 panel tolerance

[`CLAUDE.md`](../CLAUDE.md) lists "N-1 panel tolerance" as a core
invariant. The corpus supports this directly.

### 7.1 Per-call error rate

From the `events.db` event counts:

- `MODEL_RESULT` events (successful model returns): **1405**
- `MODEL_ERROR` events (model failures): **159**
- **Failure rate: ~10% per call** (159 / 1564)

Two of the five fresh runs also hit `H2H JSON parse failed for claude`
warnings (§2.5), which is a different failure class — the model
returned content but the round's schema validation failed. Roundtable's
schema-validation-failure error class is correctly modeled as a
per-model error rather than a round-level failure.

### 7.2 What this means

If Roundtable blocked on every panel member returning successfully,
**roughly 1 in 10 calls would fail and gate every round**. For a
3-model panel, that's a per-round failure probability of
1 − (0.9)³ ≈ **27%**.

N-1 tolerance turns those failures from "deliberation aborts" into
"deliberation continues with two perspectives instead of three."
Given that the manual workflow Roundtable replaces routinely operates
with whichever models happened to respond, this is the right
trade-off.

## 8. Summary: which design choices this evidence supports

| Design choice | Doc | Evidence in this doc |
|---|---|---|
| No vote-to-continue field | [`design.md` §3](design.md) | §2 (99.3% stop, 32% boilerplate, fresh runs corroborate) |
| Claude is the orchestrator, not a panel-side judge | [`decisions.md` §2](decisions.md) | §2.3 (no model can be trusted to honestly request continuation) |
| No synthesis stage | [`decisions.md` §2](decisions.md) | §4 (low cost but structurally harmful; provenance is 32% of wall) |
| No persistence | [`decisions.md` §3](decisions.md) | (Argument is privacy/UX, not in corpus — but corpus shows runs are never re-consumed) |
| Per-call timeout default 90s, max 180s | [`design.md` §2.1](design.md) | §5 (dispatch p90 = 182s; head_to_head p90 = 98s) |
| N-1 panel tolerance | [`CLAUDE.md`](../CLAUDE.md) | §7 (10% per-call error rate; 27% per-round if blocking) |
| 3-model default panel | [`design.md` §1](design.md) | §5 (latency budget), §6 (cost budget) |
| Anthropic NOT in default panel | [`design.md` §1](design.md) | §2.5 (claude's vote payload malformed in 2/5 fresh runs — Claude is best deployed as orchestrator, not panel member) |

## 9. How to reproduce or extend

To re-verify any number in this document:

1. **Internal corpus stats** — open the author's local
   `crucible-internal/runs/` directory and re-run the analysis
   script that produced the tables (see git history of this
   commit; the script is not committed because it operates on
   private prompt content).
2. **Fresh comparison runs** — re-run from the Senteron CLI:
   ```bash
   ./senteron.sh "<prompt>" --pipeline peer_review --save
   ```
   Run output lands in Senteron's `runs/` directory; the schema
   matches the corpus.
3. **Vote distribution per run** — `head_to_head_rounds[i][model].
another_round_needed` is the field. Note that some files store
`head_to_head_rounds` as a list and others as `{rounds: [...]}` with
metadata; either is valid.
4. **Stage timings** — `stage_timings.<stage>` is a number in
seconds (when populated). The `provenance` stage is the most
commonly null because it was added later in Senteron's lifecycle.

If you need to *add* new evidence (e.g., when adding a v0.2 feature
that changes the design assumptions), append a new section here
rather than editing this one — the audit trail should be append-only.

## 10. What this document is not

This document is not the design. The design is in [`design.md`](design.md).

This document is not the rationale. The rationale is in
[`decisions.md`](decisions.md).

This document is the *receipts*. If a future contributor or reviewer
asks "where did the 99.3% number come from," the answer is here, with
run IDs and verbatim quotes. If a future version of Roundtable
re-litigates a settled design question, this document is the evidence
the original decision was made against.
